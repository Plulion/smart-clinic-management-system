# Smart Clinic Management System - Schema Design

## Overview

The Smart Clinic Management System uses a hybrid database design. MySQL is used for structured and relational data such as patients, doctors, appointments, and administrators. MongoDB is used for flexible document-based data such as prescriptions, doctor notes, metadata, and medication instructions.

This design separates transactional clinic operations from flexible clinical documentation. MySQL enforces strong relationships and constraints, while MongoDB allows prescription records to evolve without requiring frequent schema migrations.

---

## MySQL Database Design

### Table: patients

| Column | Data Type | Constraints | Description |
|---|---|---|---|
| id | BIGINT | Primary Key, Auto Increment | Unique patient identifier |
| name | VARCHAR(100) | NOT NULL | Patient full name |
| email | VARCHAR(100) | NOT NULL, UNIQUE | Patient email address |
| password | VARCHAR(255) | NOT NULL | Encrypted patient password |
| phone | VARCHAR(20) | NULL | Patient contact number |
| address | VARCHAR(255) | NULL | Patient address |
| date_of_birth | DATE | NULL | Patient date of birth |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation date |

**Design notes:**
- Patient email must be unique because it is used for login.
- Passwords should be encrypted before being stored.
- Appointments should preserve historical records even if patient information changes.

---

### Table: doctors

| Column | Data Type | Constraints | Description |
|---|---|---|---|
| id | BIGINT | Primary Key, Auto Increment | Unique doctor identifier |
| name | VARCHAR(100) | NOT NULL | Doctor full name |
| specialty | VARCHAR(100) | NOT NULL | Doctor medical specialty |
| email | VARCHAR(100) | NOT NULL, UNIQUE | Doctor email address |
| password | VARCHAR(255) | NOT NULL | Encrypted doctor password |
| phone | VARCHAR(20) | NULL | Doctor contact number |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation date |

**Design notes:**
- Doctor email must be unique because doctors log into the system.
- Specialty is required so patients can search doctors by medical area.

---

### Table: admin

| Column | Data Type | Constraints | Description |
|---|---|---|---|
| id | BIGINT | Primary Key, Auto Increment | Unique admin identifier |
| username | VARCHAR(50) | NOT NULL, UNIQUE | Admin login username |
| password | VARCHAR(255) | NOT NULL | Encrypted admin password |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation date |

**Design notes:**
- Admin username must be unique.
- Admin passwords must never be exposed in API responses.

---

### Table: appointments

| Column | Data Type | Constraints | Description |
|---|---|---|---|
| id | BIGINT | Primary Key, Auto Increment | Unique appointment identifier |
| doctor_id | BIGINT | Foreign Key → doctors(id), NOT NULL | Doctor assigned to the appointment |
| patient_id | BIGINT | Foreign Key → patients(id), NOT NULL | Patient assigned to the appointment |
| appointment_time | DATETIME | NOT NULL | Appointment start date and time |
| status | INT | NOT NULL, DEFAULT 0 | 0 = Scheduled, 1 = Completed, 2 = Cancelled |
| reason | VARCHAR(255) | NULL | Reason for the visit |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation date |

**Design notes:**
- Each appointment belongs to one doctor and one patient.
- Appointment time should be validated in the application layer to prevent past appointments.
- The system should prevent overlapping appointments for the same doctor.
- Historical appointments should usually be retained for audit and clinical continuity.

---

### Table: doctor_available_times

| Column | Data Type | Constraints | Description |
|---|---|---|---|
| id | BIGINT | Primary Key, Auto Increment | Unique availability record |
| doctor_id | BIGINT | Foreign Key → doctors(id), NOT NULL | Doctor who owns this available time |
| available_time | VARCHAR(50) | NOT NULL | Available time slot, such as Monday 09:00 |

**Design notes:**
- This table supports the doctor's list of available times.
- It represents the relational storage generated by an `@ElementCollection` in the Doctor model.
